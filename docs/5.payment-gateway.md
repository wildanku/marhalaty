# Task 6: Payment Gateway, Webhooks & Task Scheduling

## 1. Context & Business Logic
This is the core financial engine. The system must support multiple payment gateways (Midtrans and Xendit) seamlessly using the **Strategy Pattern**. It also handles incoming webhooks securely (idempotency) and manages automated background tasks, such as releasing reserved merchandise stock if an RSVP payment expires.

### Key Features:
1. **Strategy Pattern:** Controllers do not interact with Midtrans/Xendit directly. They interact with a `PaymentGatewayInterface`.
2. **Dynamic Admin Fee:** Admin fees are added to the transaction dynamically based on settings, not hardcoded.
3. **Idempotent Webhooks:** Prevent double-crediting if the payment provider sends the same webhook payload multiple times.
4. **Stock Reversal (Cron):** A scheduled task to clean up expired `pending` RSVPs and return the addon stock.

## 2. Implementation Task for AI Agent
**Instruction for AI Agent:** Generate the Interfaces, Services, Webhook Controllers, and Console Commands within the `app/Domains/Shared` or respective directories to handle the payment abstraction and background jobs.

### A. Backend: The Strategy Pattern
1. **Interface (`App\Contracts\PaymentGatewayInterface`):**
   * Methods: `createInvoice(array $data): string` (Returns payment URL), `handleWebhook(Request $request): array` (Parses provider payload into standard internal format).
2. **Implementations (`App\Services\Payments\`):**
   * Create `MidtransService` and `XenditService` implementing the interface.
3. **Service Provider (`App\Providers\PaymentServiceProvider`):**
   * Bind the interface to the correct service based on `env('ACTIVE_PAYMENT_GATEWAY')`.

### B. Backend: Settings Service (Admin Fee)
1. **`App\Services\SettingService`:**
   * Create a simple service backed by Redis or a `global_settings` table to retrieve `payment_fee` (e.g., Rp 4000). Add this fee to the `gross_amount` before sending it to the Payment Gateway.

### C. Backend: Webhook Handling & Idempotency
1. **Package:** Install `spatie/laravel-webhook-client` to handle idempotency.
2. **Webhook Controller (`App\Http\Controllers\WebhookController`):**
   * Route payloads to the active Payment Service to be parsed.
   * **Database Transaction (CRITICAL):** Wrap the updates in `DB::transaction()`.
   * **Donations Logic:** If status is `paid`, update `donations` status AND increment `collected_amount` in the related `Fund` or `Campaign`.
   * **RSVP Logic:** If status is `paid`, update `rsvps` status to `paid`. If `failed`/`expired`, update status and trigger Stock Reversal.

### D. Backend: Task Scheduling (Stock Reversal)
1. **Console Command (`App\Console\Commands\ExpirePendingRsvps`):**
   * Query all `rsvps` where `status === 'pending'` and created older than 24 hours.
   * Loop through them, update status to `expired`.
   * Read the `add_ons_snapshot` JSON column. For each addon, increment the `stock_quantity` in the `event_addons` table to return the reserved stock.
2. **Kernel (`App\Console\Kernel`):**
   * Schedule this command to run hourly: `$schedule->command('rsvp:expire')->hourly();`.
